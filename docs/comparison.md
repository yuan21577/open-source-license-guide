# 主流开源许可证深度解析与对比



## 一、宽松型许可证 (Permissive Licenses)

宽松型许可证允许用户几乎无限制地使用、修改和分发，**核心特点是允许闭源衍生，商业友好**。

*   **MIT 许可证**
    *   **起源/制定者：** 麻省理工学院 (MIT)。
    *   **特点：** 极简设计，仅要求保留版权声明。缺乏专利保护。
    *   **经典案例：** Facebook 曾发布 `React` 的 `BSD+Patents` 许可证引发社区强烈抵制，最终于 2018 年妥协将其更改为 MIT 许可证，消除了开发者的专利诉讼风险。
*   **Apache 2.0 许可证**
    *   **起源/制定者：** Apache 软件基金会 (ASF)。
    *   **特点：** 相比 MIT 增加了**明确的专利授权条款**（防止专利流氓），且要求修改过的文件必须添加修改通知。极其适合企业级、大型底层开源项目（如 Kubernetes, Hadoop）。
*   **BSD 许可证**
    *   **起源/制定者：** 加州大学伯克利分校 (UCB)。
    *   **演进：** 经历了从 `4-Clause`（包含繁琐广告条款）到 `3-Clause`（保留“不得使用作者名字背书推广”条款），再到最精简的 `2-Clause` 的演变。
*   **木兰宽松许可证 (MulanPSL)**
    * **起源/制定者：** 由北京大学牵头、中国开源云联盟联合发布。**是中国首个国际认可的开源许可证。**
    
    * **特点：** 包含专利授权，提供严格的中英双语版本，契合中国法律环境，且**与 GPLv3 和 Apache 2.0 完美兼容**。  
    
        

## 二、传染型 / 著佐权许可证 (Copyleft Licenses)

Copyleft 的核心思想是：利用版权法反转版权限制，**要求衍生作品必须保持开源状态**。

*   **GPL (GNU通用公共许可证)**
    *   **起源/制定者：** 自由软件基金会 (FSF, Richard Stallman)。
    *   **特点：** 强传染性。任何包含、动态链接、静态链接 GPL 代码的程序，其分发时都必须以 GPL 协议开源。
    *   **演进：** GPLv3 (2007) 相比 v2，增加了反 DRM (数字版权管理) 限制，并明确禁止专利诉讼。
*   **LGPL (GNU宽通用公共许可证)**
    
    *   **特点：** 弱传染性。专为库文件（Library）设计。**如果是动态链接 (Dynamic Linking) LGPL 库，主程序无需开源**；但如果是静态链接或修改了库本身，则受传染性约束。
*   **MPL (Mozilla公共许可证)**
    
    * **特点：** **文件级传染性**。只要不修改 MPL 原文件，将其与专有代码文件组合成项目，专有代码无需开源。非常适合与商业软件结合。
    
      

## 三、判定矩阵

### 1. 许可证核心属性矩阵

本表聚焦于开发者最关心的**商业合规、传染性范围与专利免责**维度。（整合了 MIT、Apache、GPL 系列及国产木兰协议）

| 许可证维度        |     MIT / BSD      |    Apache 2.0    | 木兰 MulanPSL v2 |     LGPL (弱传染)     |    GPL-3.0 (强传染)    |     MPL (弱传染)      |
| :---------------- | :----------------: | :--------------: | :--------------: | :-------------------: | :--------------------: | :-------------------: |
| **商业闭源分发**  |       ✅ 允许       |      ✅ 允许      |      ✅ 允许      | ✅ 允许 (仅限动态链接) |     ❌ **严格禁止**     | ✅ 允许 (需文件级隔离) |
| **传染性级别**    |     ❌ 无传染性     |    ❌ 无传染性    |    ❌ 无传染性    |    ⚠️ **库级传染**     |    ⚠️ **项目级极强**    |   ⚠️ **文件级传染**    |
| **专利明文授权**  |      ❌ 无明文      |       ✅ 有       |       ✅ 有       |         ✅ 有          |          ✅ 有          |         ✅ 有          |
| **漏洞免责声明**  |     ✅ 概不负责     |    ✅ 概不负责    |    ✅ 概不负责    |      ✅ 概不负责       |       ✅ 概不负责       |      ✅ 概不负责       |
| **商标/背书限制** |  ❌ 禁止背书(BSD)   |    ❌ 禁止使用    |      未提及      |        未提及         |         未提及         |        未提及         |
| **典型适用场景**  | 小型工具、前端框架 | 大型基建、云原生 | 中国合规环境项目 | 底层依赖库(如FFmpeg)  | 独立模块、操作系统内核 |  商业与开源混合项目   |

### 2. 许可证兼容性判定矩阵 (SCA合规左移核心)

在实际的软件成分分析 (SCA) 中，判断引入的第三方包是否会引发“合规冲突”至关重要。下表展示了不同许可证代码组合时的兼容情况：

| 引入的第三方包许可证 |    能否并入 MIT 项目？     | 能否并入 Apache 2.0 项目？ | 能否并入 GPLv3 项目？ | 能否作为闭源商业软件发布？ |
| :------------------- | :------------------------: | :------------------------: | :-------------------: | :------------------------: |
| **MIT / BSD**        |         ✅ 完美兼容         |         ✅ 完美兼容         |        ✅ 兼容         |           ✅ 允许           |
| **Apache 2.0**       |         ✅ 完美兼容         |         ✅ 完美兼容         |        ✅ 兼容         |           ✅ 允许           |
| **LGPL**             |      ✅ (仅限动态链接)      |      ✅ (仅限动态链接)      |        ✅ 兼容         |      ✅ (仅限动态链接)      |
| **MPL**              |          ❌ 不兼容          |          ❌ 不兼容          |        ✅ 兼容         | ✅ (未修改MPL原文件前提下)  |
| **GPLv3**            | ❌ (会感染并强制开源主项目) | ❌ (会感染并强制开源主项目) |      ✅ 完美兼容       |       ❌ **绝对禁止**       |

> **💡 兼容性判定原则总结：**
>
> 1. **宽松型 (Permissive)** 许可证之间通常相互兼容，且对商业闭源极其友好。
> 2. **传染型 (Copyleft)** 许可证具有“单向吞噬性”。如果你在项目中哪怕只用了一行 GPL 代码，你的整个项目也必须以 GPL 协议开源（即所谓的“传染性”）。
> 3. 企业在商业开发中，应利用 SCA 工具严格排查并隔离 GPL 协议组件。



## 四、真实合规风险案例分析

> 以下案例均来自开源史上真实发生的许可证争议事件，是理解许可证合规重要性的最佳素材。

### 案例一：React 的 BSD+Patents 风波（2017）

**事件经过：**
Facebook 曾为其前端框架 `React` 使用了一种非标准的 `BSD + Patents` 许可证。该许可证附加了一项专利条款：**如果你（或你的公司）起诉 Facebook 任何专利侵权，你使用 React 的许可将自动终止。**

这意味着使用 React 的企业，实际上放弃了对 Facebook 的专利诉权。Apache 软件基金会（ASF）随后将该许可证列为不可信任（Category X），禁止旗下所有 ASF 项目使用 React。

**结果：**
面对社区的强烈抵制，Facebook 于 2017 年宣布将 React 重新授权为标准 MIT 许可证，消除了开发者的专利诉讼风险。

**合规启示：** 许可证中的"附加条款"可能隐藏重大法律风险，SCA 工具不仅要识别许可证名称，还需解析附加条款。



### 案例二：Elasticsearch 的协议突变（2021）

**事件经过：**
Elasticsearch 是业界最流行的搜索引擎之一，长期使用 Apache 2.0 许可证。2021 年，其母公司 Elastic 突然宣布将许可证更改为 **SSPL（Server Side Public License）和 Elastic License 2.0**。

SSPL 是 MongoDB 发明的一种许可证，要求：**任何将该软件作为云服务对外提供的公司，必须将整个服务栈（包括自研代码）全部开源。** OSI（开源促进会）明确拒绝认定 SSPL 为开源许可证。

**结果：**
AWS 等云服务商直接 Fork 了 Elasticsearch 的最后一个 Apache 2.0 版本，创建了独立的开源项目 **OpenSearch**，并由 AWS 持续维护至今。

**合规启示：** 上游项目随时可能改变许可证，企业应当在 SBOM 中锁定依赖版本，并在升级前进行许可证变更审查。



### 案例三：MySQL 的双重许可证困局

**事件经过：**
MySQL 数据库采用 **GPL-2.0 + 商业授权** 的双重许可模式。在 GPL 协议下，任何将 MySQL 集成进商业产品并分发的企业，理论上必须将整个产品以 GPL 开源——这对商业软件是不可接受的。

因此，MySQL（现为 Oracle 所有）同时提供商业许可证，企业需付费购买才能在闭源商业产品中合法使用。

**结果：**
这一模式催生了 `MariaDB`（MySQL 创始人离开 Oracle 后创立的纯开源分支），以及大量企业选择迁移至 PostgreSQL（BSD 许可证，完全商业友好）。

**合规启示：** GPL 许可证的"传染性"对商业软件影响极大。`mysqlclient`（Python 的 MySQL 驱动）同样是 GPL-2.0，在商业项目中引入时须特别注意，可考虑改用 `PyMySQL`（MIT 许可证）作为替代。
